# 审查模式

用于维护现有知识库：检查矛盾、重复、断链、过期页面、孤儿页面、scrap/dropzone 聚类，以及从长期 log 中提炼 self 信号。

## Step 1：定审查范围

不需要先读全部规则。先根据用户请求和目录结构确定范围：

- 只查某页；
- 只查 `wiki/`；
- 只查 `self/`；
- 只查 `raw/dropzone/` / `wiki/scrap/`；
- 全库健康检查。

如果用户未指定范围，默认先读 `index.md`、近期 `log.md`、`pending.md`，再决定下一步要查哪些目录。

## Step 2：按范围读取规则

按实际范围读取：

- 查 root 文件时读 `schemas/root.md`。
- 查 YAML/状态/日期时读 `schemas/common.md`。
- 查 wiki 页面时读 `references/wiki.md` 和 `schemas/wiki.md`。
- 查 self 页面或整理画像时读 `references/self.md` 和 `schemas/self.md`。
- 查 raw/dropzone 或附件指针时读 `references/raw.md` 和 `schemas/raw.md`。

不要为没涉及的层读取规则。

## Step 3：确定性检查

逐项检查当前范围内的问题：

- YAML 字段缺失或受控词表漂移。
- `index.md` 缺条目或链接失效。
- `log.md` 缺记录、记录不可解析或日期不清。
- 页面没有来源、时间线或 updated 日期。
- 重复页面、相似标题、同源重复。
- 过期或强实时页面。
- 页面之间的冲突。
- `wiki/scrap/` 与 `raw/dropzone/` 是否可聚类升级。
- 非 markdown 素材是否缺少 markdown 指针文件。

低风险修复可直接做；会改变语义、合并、删除或提升 self 结论时先问用户。

## Step 4：self 升维

只有用户要求整理画像，或审查范围包含 self/log 行为信号时，才读取 `references/self.md` 和 `schemas/self.md`。

读近期 `log.md` 和 `self/observations.md`：

- 反复出现的主题可能是注意力信号。
- 反复出现的做法可能是工作流偏好。
- 反复出现的评价标准可能是价值观或决策方式。
- 过期的 `self/now.md` 项目应移到历史区，不直接删除。

只有跨时间重复、证据足够、对未来个性化有用时，才建议写入 self。模糊结论进入 `pending.md` 等确认。

## Step 5：清理 pending.md

每次审查模式都做这一步，不需要用户额外要求——这是 `pending.md` 唯一的清理入口。

`pending.md` 不是档案：它只是提醒用户接下来要看什么，历史已经由 `log.md` 和 git 记录。打勾不是"已读"，是用户在说"这条划掉了"（已经搞定/已经放弃/已经确认/不重要了）。

这是机械类操作（按 `[x]` / `[ ]` 这个明确标记过滤，不需要理解条目内容），优先用 blind 脚本/命令行执行，不要先把全文 Read 进 agent 上下文再人工挑：

1. 用一次性脚本（grep/sed/awk/python 均可）扫描 `pending.md`，删除所有 `- [x]` 开头的条目（包括它下面缩进的子说明，整块一起算），不挪到 `log.md`、不存档、不另写摘要——勾选本身就是用户给的删除授权，不需要再向用户确认一次。
2. 未打勾（`- [ ]`）的条目原样保留，不因为审查模式自己判断"这条不重要"就删除或打勾——那是用户的判断。
3. 验证用计数和 diff（`wc -l`、`grep -c '\[x\]'`、`git diff --stat`），不要为了"确认做对了"把全文读出来或打印出来——这一步本身不需要语义判断，结果对不对从计数和 diff 就能确认。
4. 清理完报告删除了多少条、剩余多少条；如果某个区块（待确认/数据缺口/想法）被清空，保留空标题，不删除区块本身。

## Step 6：整理 log.md

每次审查模式都做这一步。`log.md` 服务于"快速翻看最近做了什么"，精确的版本历史由 git 承担，不需要 `log.md` 逐字还原每一处改动。

1. **当月（含本月）条目**：保持颗粒度，一条操作一条记录，不做月度归档——当月内容是最常被翻看的。但如果发现同一 topic 在几小时内被拆成了好几条独立记录（本该在写入时就合并，见 `schemas/root.md`），现在可以合并成一条，按时间顺序保留各小步骤。
2. **非当月（上月及更早）条目**：按自然月分组，每个自然月最多合并成一条 `## [归档时间] 归档 · <YYYY-MM> 月度汇总`，内部按时间顺序用子 bullet 列出当月做过的关键操作（保留可溯源事实：页面名/数量/topic，措辞可以精简但事实不能丢）。某个月如果已经归档过，后续没有新条目就不用再动；如果该月又有条目混入（不太可能但理论上存在），追加进已有的归档条目而不是新建。
3. 归档**不是删除**，仍然遵守"日志绝不删除"硬规则——只是把多条记录收进一条、精简措辞。不确定某条该不该精简，宁可多保留细节。
4. 这一步本质是按日期分组+合并，机械成分高，可以用脚本辅助分组和排序，但"哪些细节可以精简、哪些必须保留"仍需要 agent 判断，不要让脚本自己决定删什么措辞。
5. 完成后报告：归档了哪几个月、从多少条压缩到多少条。

## Step 7：收尾

如需写入，再读取 `schemas/root.md`：

- 已做的低风险修复写入 `log.md`。
- 待确认事项写入 `pending.md`。
- 如果建议新增类型或新规则，说明理由，并建议更新对应 reference/schema。

## 输出

输出审查报告：

- 发现：按严重程度列问题。
- 已处理：列低风险修复。
- 需要决定：列需要用户确认的合并、删除、self 推断。
- 下一步：建议下一轮维护。
