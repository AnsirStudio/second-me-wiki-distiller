# 审查模式（review）

维护现有知识库:补 capture/落库留下的粗糙、清遗留、升 self。**永不全量读库**——靠三件事:游标(只看变化的)、机械/判断分层、按切片。

## 核心原则

- **review 永不一次性读全库**。库大了 100M 上下文也不够。只处理:游标之后变化的、或某个切片、或被 flag 的小集合。
- **机械 vs 判断分开**:能用脚本 blind 跑的(计数/grep/sed/index 同步/删打勾)绝不读全文;只有需要理解语义的(判重、评估该删该留、聚类升级)才读那几页。
- 低风险修复直接做;改语义、合并、删除、升 self 结论前先问用户。

## Step 1：定范围 + 读游标

- 游标存库根 `review-state.md`(没有就建):`last_reviewed_at: YYYY-MM-DD HH:MM`(本机本地时间,时区由使用方项目 CLAUDE.md 指定)+ 一小段手动 flag 队列。
- **"上次到哪、哪些要看" = log 就是变更流**:`grep '^## \[' log.md` 取 `last_reviewed_at` 之后的条目,它们提到的 `[[页]]` 就是"自上次以来动过的页" = 本轮判断层的默认输入。不用另维护清单。
- 按用户请求定切片(不指定就默认"游标之后变化的 + 跑一遍全库机械层"):
  - 只查某页 / 某类型(如只 entities)/ 某簇(某创作者的一组页)/ `verified:false` / scrap+dropzone / 游标之后变化的 / 全库健康(= 机械层扫全库,判断层仍只看 flag)。

## Step 2：机械层（脚本 blind,成本与库大小无关,可每次跑）

用 grep/sed/一次性脚本 + 计数/diff 验证,**不读全文**:

1. **index.md 同步**:按 `wiki/` 各子文件夹扫每个文件 frontmatter 的 `title`+`tldr`,重新生成/校验 index 对应行——`title` 与文件名不同写 `[[文件名|title]]`、相同写 `[[文件名]]`(index 用裸名);summary 不进 index;`-全部作品` 台账不进;章节扁平不分组。新增补行、删除去行、`tldr` 改了同步。
2. **pending 清打勾**:删所有 `- [x]`(含其下缩进子说明)。注:integrate 每批已清一次,这里是保底完整清理。
3. **log 月度归档**:非当月条目按自然月合并成一条 `归档 · <YYYY-MM>`(不删除、只精简合并,保留页名/数量/topic);当月保持逐条。若同一 topic 几小时内被拆成多条,合并成一条。
4. **扫质量旗标**(grep,只报不改,进 flag 队列):断链 `[[不存在的页]]`、孤儿页(无入链)、缺字段(无 `tldr`/来源/`updated`——**排除 `-全部作品` 台账**,它们不进 index、不需完整 frontmatter)、被反引号包住的 `[[]]`、正文里残留的 `[[wiki/类型/` 全路径(正文该裸名)。
5. **清遗留「编辑日志」时间线**:删 entity 页里 `## 时间线` / `## 版本里程碑` 这类**页内编辑日志**段(内容形如 `### [日期] vN · 新增X`)——改动史由 git/log 承担,页内不重复记。**保留真·内容时间线**:段名是 `功能演进`/`发展时间线`/`功能/里程碑时间线`、或内容讲的是**主体自己的历史**(产品功能演进、话题发酵)而非"这页第几版改了啥"。拿不准的先 flag、别删。

## Step 3：判断层（agent 读,只读被 flag / 切片内的小集合，不读全库）

1. **重复/相似页**:标题、别名、同源。真重复才合并——合并=择优融合 + 互链 + 保留双方来源,不是简单删一个。
2. **冲突页**:同一事实两页说法不一 → 保留双方来源和日期、标冲突,不静默覆盖。
3. **scrap / dropzone 聚类升级**:反复出现的碎片能否升成正式 concept/method/entity。
4. **过期强实时页**:`valid_as_of` 老、`volatility: volatile` 的,标状态或提示回查。
5. 逐个补机械层报的旗标(断链/孤儿/缺字段)。

## Step 4：self 升维（仅当范围含 self/log 行为信号，或用户要整理画像）

读近期 `log.md` + `self/observations.md`(读 `references/self.md` 和 `schemas/self.md`):
- 跨时间反复出现的主题/做法/评价标准 → 可能是注意力信号 / 工作流偏好 / 价值观。
- 只有跨时间重复、证据够、对未来个性化有用,才建议写入 self;模糊结论进 `pending.md` 等确认。过期的 `self/now.md` 项移历史区、不直接删。

## Step 5：收尾

- 低风险修复写一条 `review` log(新格式:本批总况 + 逐行页名 `[[wiki/...]]` 全路径可点;时间用本机本地时间)。
- 需用户确认的合并/删除/self 结论 → `pending.md`。
- **推进游标**:把 `review-state.md` 的 `last_reviewed_at` 更新到本轮时间;flag 队列里处理掉的划掉。
- 若发现该新增类型/规则,说明理由并建议改对应 `references/`/`schemas/`。

## 输出

审查报告:发现(按严重程度)、已处理(低风险修复)、需要决定(合并/删除/self 推断)、下一步(建议下个切片)。
