# Root Schema

适用于知识库根目录的 `index.md`、`log.md`、`pending.md`。

## index.md

条目少于 1000 时，`index.md` 是主导航。

```markdown
# 目录

## 概念

- [[example]] - 一句话 tldr。

## 实体

## 话题

## 方法

## 小技巧

## GitHub

## 对比

## 引用

## 碎片

## 自我记忆
```

超过 1000 条时，可以建议增加本地搜索、BM25 或向量层，但仍保留 `index.md`。

`index.md` 只索引"编译后的知识"层（concept/entity/topic/method/tip/github/comparison/quote），**不索引 summary**。summary 是 raw 的前引速读，不是核心知识页（见 `schemas/wiki.md`），把它也列进 index.md 会让根导航随来源数量线性膨胀、稀释"先扫 tldr"的作用。summary 的可达性由它所属的 entity/concept 页面的"已蒸馏作品"或"关联"链接保证：检索路径是 index → entity/concept → summary（如果想看更细）→ raw（如果想看原文），不需要 index.md 兜底。

`index.md` 每条只放跟该页面 frontmatter `tldr` 等量的一句话，**不要**在 index 里另起一份比 tldr 更详细的描述，尤其不要把"已蒸馏作品"逐条列出来贴进 index——entity 页更新了多少条来源、覆盖了哪些主题，这些细节属于 entity 页本身（或它的台账文件），index.md 只需要这条 entity 现在"是什么"。entity 页的 tldr 改了，index.md 对应行跟着同步；不要各自独立生长出两份不同详细程度的描述。

`index.md` 的结构必须严格 1:1 镜像 `wiki/` 的实际文件夹结构：章节标题只能是受控词表里的顶层类型（概念/实体/话题/方法/小技巧/GitHub/对比/引用/碎片/自我记忆），且每个章节内部是**扁平列表**，不允许任何额外的主题/平台/厂商分组——既不能开同级 `##` 标题（例如"Google AI 产品栈""YouTube/内容创作者"），也不能在标题下加 `###` 子分组。`wiki/entities/` 在磁盘上就是一个没有子文件夹的扁平目录，index.md 里对应的"实体"章节也必须是扁平的，不要凭语义臆造一层 disk 上不存在的分类——想按主题浏览，靠每个实体页自己的 `related`/关系图互链，不靠 index.md 充当二级分类法。

`index.md` 里的内部链接**不带文件夹路径**，直接写文件名（`[[example]]`，需要别名时 `[[example|显示名]]`）——章节标题已经说明了类型，再写一遍 `wiki/entity/` 前缀纯属重复，只会让每一行变长。这条只对 `index.md` 生效；wiki 页面之间互相引用仍按 `references/wiki.md` 的规则带文件夹路径，避免跨上下文引用时产生歧义。

## log.md

操作日志只追加，记录库发生了什么。**目标是让用户扫一眼就懂这批干了什么、基本不用打开每个页**——所以正文要可扫读，每个页名都用 `[[]]` 可点跳转。

### 格式（标题保留，正文按下面结构）

```markdown
## [YYYY-MM-DD HH:MM] ingest · <层> · topic=<主题> · +N summary +M concept… — <标题>

**本批**：处理 X 条（raw 已移 X → `raw/socialmedia/<作者>/`）；新建 summaries Y；重复 Z（列出哪几条，已归档 `-duplicate`、未产新知识）；空/无转录 W。

**新建页 ×N**
- [[wiki/concepts/示例]] — 一句话「是什么」
- [[wiki/methods/示例]] — 一句话「是什么」

**更新页 ×N**
- [[wiki/entities/示例]] — 补了什么（一句话）

**pending 新增 ×N**
- 一句话缺口（关联 [[wiki/…]]）

**备注**：跳过 / 冲突 / 低置信（有才写）。
```

### 写入规则

- **每个页名都用 `[[wiki/<类型>/<文件名>]]` 包裹**（带文件夹路径），新建、更新、pending 关联页都要——这样在 Obsidian 里能点开。**别只用反引号 `名字`**，那点不动。
- **一页一行 + 一句话概括**：用户扫读就懂，基本对就不必打开原页；有遗漏或感兴趣再点进去。
- **本批 intro 先给总况**：几条、raw 移了几条、summaries 几条、有没有重复 / 空内容等情况。
- **时间戳**：见下方"时间戳"，必须带 `HH:MM`，只写日期是错的。
- **永远插在文件最顶部**（legend 块之后、既有记录之前），严格时间倒序；legend 只在最初写一次。
- **几小时内的同类小操作并进最近那条**，不新开标题；明显不同（不同 topic / 跨小时）才各自开新记录。
- 并行编排下，这条合并 log 由 **integrate 一次性写**（不是每条各写）。

### 时间戳（重要）

- **一律用本机（真机）本地时间，绝不用沙箱 / 服务器时间**——沙箱通常是 UTC，会比本机快好几个小时。
- 取时间：`TZ='<库时区>' date '+%Y-%m-%d %H:%M'`（库时区由使用方项目的 CLAUDE.md 指定），或直接从真机 `date` 取。

记录写入、归档、删除、合并、审查、重要结构升级。纯检索不写 log。月度归档（非当月条目按月合并）由审查模式负责，见 `modes/review.md` Step 6。

## pending.md

`pending.md` 不是档案，是给用户本人看的提醒清单——库的历史已经由 `log.md` 和 git 承担，`pending.md` 只负责"提醒用户接下来要看什么"。它应该保持小而活跃，不是越长越好。

```markdown
# pending · 待确认 / 数据缺口 / 想法

> 这不是历史记录，历史在 log.md 和 git 里。打勾 = 划掉，审查模式会把所有打勾条目直接删除，不留痕。

## 待确认

- [ ] `YYYY-MM-DD` [[页面]] - 需要确认什么。

## 数据缺口

- [ ] `YYYY-MM-DD` 缺口说明，关联页面/来源。

## 想法

- [ ] `YYYY-MM-DD` 想法一句话，可能落点。
```

写入规则：

- 三个区块下的**每一条都要带 `- [ ]` 复选框**，包括数据缺口和想法——不是只有"待确认"才有勾。
- AI 只新增未勾选条目，**不替用户打勾**，也不能因为自己判断"这条不重要"就直接删掉它——那是用户的判断，只有用户自己勾。
- 例外：如果 AI 后续的工作已经实际解决了某条（查到了答案、做完了对应功能、确认了事实），可以直接移除这条（不需要先打勾再删），因为这是工作完成、不是"我觉得不重要"。
- **打勾条目清理有两个入口**：① `integrate`（落库）每批写新缺口前先 blind 删一遍 `- [x]`（见 `modes/integrate.md`）——这样 pending 在两次 review 之间不会堆长；② `modes/review.md` 是保底的完整清理入口。两处都是「扫到打勾的（含其下嵌套子说明）直接删除，不挪 log、不存档」——勾选本身就是用户给的删除授权，不需再确认。
